# 테넌트 격리

## 개요

SaaS 플랫폼에서 테넌트(조직/워크스페이스) 간 데이터와 리소스를 격리하는 전략을 정의합니다. 격리 수준 선택, 접근 통제, 리전 데이터 레지던시를 포함합니다.

## 격리 전략

| 전략 | 격리 수준 | 비용 | 적합 대상 |
|------|----------|------|----------|
| DB 분리 | 최고 | 높음 | 엔터프라이즈 고객, 규제 산업 |
| 스키마 분리 | 높음 | 중간 | 중간 규모 고객, 커스텀 요구 |
| 행 수준 (RLS) | 보통 | 낮음 | 다수 소규모 테넌트, 빠른 확장 |

### 전략 선택 기준

| 기준 | DB 분리 | 스키마 분리 | 행 수준 |
|------|---------|-----------|---------|
| 테넌트 수 | ~100 | ~1,000 | 무제한 |
| 커스텀 스키마 | 가능 | 가능 | 불가 |
| 백업/복구 단위 | 테넌트별 | 테넌트별 | 전체 |
| 마이그레이션 | 테넌트별 순차 | 테넌트별 순차 | 전체 일괄 |
| 프로비저닝 속도 | 느림 (DB 생성) | 중간 (스키마 생성) | 빠름 (행 삽입) |

## 행 수준 보안 (RLS) 규칙

모든 테넌트 데이터 테이블에 적용하는 규칙:

| 규칙 | 설명 |
|------|------|
| tenant_id 필수 컬럼 | 모든 비즈니스 테이블에 tenant_id NOT NULL |
| RLS 정책 적용 | DB 레벨에서 tenant_id 필터 강제 |
| 애플리케이션 이중 검증 | 쿼리 파라미터와 리소스 tenant_id 일치 확인 |
| 인덱스 | 모든 조회 쿼리에 tenant_id 포함 복합 인덱스 |

## 테넌트 컨텍스트 전파

```
HTTP 요청 → 인증 미들웨어 (JWT에서 tenantId 추출)
  → 요청 컨텍스트에 tenantId 설정
    → 서비스 레이어 (tenantId 전달)
      → 리포지토리 (쿼리에 tenant_id 필터 적용)
        → DB (RLS 정책으로 이중 방어)
```

| 레이어 | 역할 |
|--------|------|
| 인증 | JWT 토큰에서 tenantId 클레임 추출 |
| API | 요청 리소스의 tenantId와 인증 tenantId 일치 검증 |
| 서비스 | 모든 도메인 연산에 tenantId 전달 |
| 리포지토리 | 모든 쿼리에 WHERE tenant_id = ? 조건 |
| DB | RLS 정책으로 누락된 필터 방어 |

## 테넌트 상태 머신

```
provisioning → active ⇄ suspended → terminated
  ↓              ↓                      ↑
  terminated     terminated ←───────────┘
  (실패 롤백)    (자진 해지)
```

| 상태 | 설명 | 전이 조건 |
|------|------|----------|
| provisioning | 테넌트 생성 중 (스키마/데이터 시딩) | 가입 + 결제 정보 등록 |
| active | 정상 운영 | 프로비저닝 완료 |
| suspended | 일시 정지 | 결제 실패, 정책 위반, 트라이얼 만료 |
| terminated | 해지 (데이터 보존 기간 진입) | 자진 해지 또는 정지 기한 초과 |

### 허용 전이

| From | To | 조건 |
|------|----|------|
| provisioning | active | 스키마 생성 + 시딩 완료 |
| provisioning | terminated | 프로비저닝 실패 (롤백) |
| active | suspended | 결제 실패 / 정책 위반 / 트라이얼 만료 |
| active | terminated | 사용자 자진 해지 |
| suspended | active | 결제 재개 / 시정 완료 |
| suspended | terminated | 정지 30일 초과 또는 시정 미이행 |

## 역할 기반 접근 통제 (RBAC)

### 역할 계층

| 역할 | 범위 | 핵심 권한 |
|------|------|----------|
| Super Admin | 플랫폼 전체 | 모든 테넌트 관리, 시스템 설정 |
| Tenant Admin | 테넌트 내 | 멤버 관리, 설정, 과금 |
| Manager | 테넌트 내 | 프로젝트/팀 관리, 멤버 초대 |
| Member | 테넌트 내 | 리소스 CRUD |
| Viewer | 테넌트 내 | 읽기 전용 |

### 권한 격리 규칙

| 규칙 | 설명 |
|------|------|
| 테넌트 범위 | Tenant Admin 이하 역할은 자신의 테넌트만 접근 |
| 크로스테넌트 금지 | 역할 권한과 무관하게 타 테넌트 리소스 접근 불가 |
| Super Admin 감사 | Super Admin의 테넌트 접근은 감사 로그 필수 기록 |

### 멤버 초대 플로우

```
Admin이 이메일 초대 → 초대 토큰 생성 (24h 유효)
  → 수신자 수락 → 회원가입(미가입 시) → 역할 할당 → 테넌트 멤버 등록
```

### API 키 관리

| 규칙 | 설명 |
|------|------|
| 테넌트 스코프 | API 키는 특정 테넌트에 바인딩 |
| 권한 범위 | API 키별 허용 퍼미션 설정 (최소 권한) |
| 만료 정책 | 기본 1년, 갱신 필요 |
| 회전 | 신규 발급 → 기존 키 유예 기간(7일) → 폐기 |
| 해시 저장 | API 키 원문 저장 금지, SHA-256 해시만 저장 |

## 리전 데이터 레지던시

| 규칙 | 설명 |
|------|------|
| 리전 선택 | 테넌트 생성 시 데이터 저장 리전 선택 (이후 변경 불가) |
| 리전 간 이동 금지 | 테넌트 데이터는 선택한 리전에만 저장 |
| GDPR 역외 이전 | EU 테넌트 데이터는 적정성 결정국 또는 SCC 없이 역외 이전 금지 |
| 리전별 격리 | 리전 간 DB 쿼리/API 호출 차단 |

## 참고사항

- 테넌트 상태 변경은 반드시 감사 로그 기록
- 프로비저닝 실패 시 생성된 리소스 전체 롤백
- suspended 상태에서는 읽기만 허용 (쓰기/삭제 차단)
- terminated 후 데이터 보존 기간은 `data-lifecycle.md` 참조
- 구독 상태와 테넌트 상태의 관계: 구독 canceled → 유예 후 테넌트 suspended
